## 3. Documentation

### A. Architecture & Component Hierarchy
SkillSwap follows a microservice-inspired monolithic architecture with a React Single Page Application (SPA) on the frontend and an Express.js API Gateway on the backend.

**Component Hierarchy (Frontend):**
- `App (Router)`
  - `AuthProvider` (Context API for State Management)
  - `AppLayout`
    - `Sidebar` & `Navbar`
    - `LoginPage` / `RegisterPage`
    - `DashboardPage` (Recharts & Stats)
    - `SkillsPage` (Grid & CRUD Modals)
    - `SessionsPage` (Status management)
    - `ProfilePage`
    - `AdminPage` (User Management)

**Assumptions:**
1. Users register with an institutional email (e.g., `@bits.edu`) which implicitly verifies their campus affiliation.
2. The initial matching algorithm relies on exact or partial text matching for tags and categories rather than a machine-learning recommendation system.
3. Sessions default to "Online" unless a specific campus location is provided.

### B. Database Schema (MongoDB/Mongoose)
The database consists of three primary collections:
1. **Users:** Stores authentication details, role (`student`, `mentor`, `admin`), bio, ratings, and references to skills offered/wanted.
2. **Skills:** Stores skill details (name, category, proficiency level), searchable tags, and a reference to the user offering it.
3. **Sessions:** Manages the lifecycle of a mentoring request (`pending` → `accepted` → `completed`), tracking the mentor, learner, scheduled date, and post-session review/rating.

*(Detailed DB diagrams and API specifications are available in the `/docs` folder of the GitHub repository).*

### C. API Documentation Summary
The RESTful API provides endpoints protected by JWT authentication:
- **Auth Service:** `/api/auth/register`, `/api/auth/login`, `/api/auth/profile`
- **User Service:** `/api/users` (CRUD with pagination and role filtering)
- **Skill Service:** `/api/skills` (CRUD, category filtering, full-text search)
- **Session Service:** `/api/sessions` (Scheduling, status updates, ratings)

## 4. AI Usage Log and Reflection Report

### Approach
The entire SkillSwap application was built using a hybrid **AI-assisted development workflow** utilizing Claude (Anthropic) and Cursor AI.

### Which AI Tools Were Used and How
- **Claude (Anthropic):** Used primarily for architecture design, complex algorithm structuring (like the dashboard statistics aggregation pipeline), and debugging React state issues.
- **Cursor IDE:** Used for inline code generation, rapid component scaffolding, and generating the CSS design system (glassmorphism tokens).

### Example Prompts Used
1. *"Design a full-stack web application architecture for a campus skill exchange platform with React frontend, Node.js backend, and MongoDB database."*
2. *"Create a session controller with CRUD operations, status management (pending → accepted → completed), and a review system with rating aggregation."*
3. *"Create an animated React sidebar with Framer Motion, collapsible state, and role-based menu items for student, mentor, and admin."*

### Parts Generated by AI vs. Manually Coded
- **AI-Generated (~70%):** Database schema blueprints, Express route boilerplate, React component structures, Recharts integration, and the base CSS design system.
- **Manually Coded (~30%):** Custom error handling logic, JWT security implementation, adjusting UI/UX colors to achieve the premium glassmorphism feel, resolving port conflicts (e.g., macOS AirPlay on port 5000), and integration testing.

### Reflection: Did AI help or hinder understanding?
AI significantly accelerated development (reducing a multi-day project to hours) while maintaining code comprehension. It helped establish standard, secure patterns (like bcrypt hashing and JWT handling) that would otherwise require tedious boilerplate. However, over-reliance can hinder understanding if the generated code isn't reviewed line-by-line. Treating AI as a "collaborative pair-programmer" rather than an "auto-generator" was the key to maintaining control over the business logic.

### What issues did you encounter integrating AI output?
1. **Context Fragmentation:** The AI occasionally forgot previous architectural decisions (e.g., assuming a `uuid` string instead of a MongoDB `ObjectId`).
2. **CSS Specificity Overlaps:** The generated glassmorphism CSS had conflicting z-indexes which caused modals to render behind the sidebar.
3. **Environment Assumptions:** The AI assumed default ports (Port 5000) which conflicted with local macOS services, requiring manual port reconfiguration to 5001.

### What did you learn from debugging AI-generated code?
I learned that AI is excellent at generating "happy path" code but often misses critical edge cases. Debugging AI code reinforced the importance of understanding the underlying framework (React lifecycle hooks, Express middleware chain). It taught me to incrementally test AI outputs rather than pasting large blocks of code, and that human intuition is still strictly required to map technical outputs perfectly to the intended business requirements.
